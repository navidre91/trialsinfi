# Pipeline Website Sync Implementation

This document explains how the current pipeline-to-website integration works, why it is split between GitHub Actions and Namecheap cPanel, and what the next person should know before changing it.

It is intended to be standalone. A new engineer should be able to read only this file and understand:

- where the website data actually lives
- how the Python pipeline feeds the PHP website
- what is deployed to Namecheap
- what is run on GitHub
- which files are important
- how to test, troubleshoot, and operate the sync

## 1. High-Level Architecture

The implementation is intentionally split into two environments:

1. GitHub Actions runs the Python pipeline weekly on a modern Python runtime.
2. Namecheap shared hosting serves the PHP website and imports the generated catalog JSON.

This split exists because the website runs fine on Namecheap shared hosting, but the pipeline does not belong there:

- Namecheap shared hosting has PHP and SQLite support, which is enough for the website.
- The current production-ready pipeline uses newer Python features and should run on Python 3.11+.
- The Namecheap shared host available during implementation exposed Python 3.6.8, which is too old for the current pipeline code.

So the operational model is:

- GitHub Actions generates `website_trials.json`
- GitHub uploads that JSON to Namecheap over SSH
- Namecheap runs the PHP importer
- The website reads the updated catalog through `api/trials.php`

## 2. What The Website Actually Uses

The website does not use the Python pipeline output directly.

The live catalog for the site is `data/trials.json`.

That file is the website-facing source of truth for trial content. The PHP backend normalizes and serves it through the API.

Important runtime pieces:

- `data/trials.json`
  The live website catalog.
- `api/bootstrap.php`
  Backend normalization, validation, catalog loading/writing, metadata handling, and SQLite bootstrap.
- `api/trials.php`
  Public/admin trial API used by the frontend.
- `data/physician_forum.sqlite`
  SQLite database used for auth/community/forum data.

The frontend now reads:

- `trials`
- `metadata`
- `count`

from `api/trials.php`, and only falls back to `data/trials.json` if the API fails.

## 3. What The Pipeline Produces

The pipeline lives in:

- `knowledgecenter/production_ready_pipeline/`

The key output for the website is:

- `website_trials.json`

This is generated by:

- `knowledgecenter/production_ready_pipeline/website_export.py`

The pipeline runner that produces this during a normal run is:

- `knowledgecenter/production_ready_pipeline/gu_pipeline.py`

The exporter converts pipeline study/site records into a website-native JSON payload:

- top-level `metadata`
- top-level `trials`

The metadata currently includes:

- `exportType`
- `exportedAt`
- `lastSyncAt`
- `pipelineVersion`
- `sourceRun`
- `sourceRunDir`
- `trialCount`
- `institutionCount`

The trial records include both:

- legacy fields the existing website expected
- richer pipeline fields such as:
  - `cancerTypes`
  - `phaseRaw`
  - `diseaseSettingPrimary`
  - `diseaseSettingAll`
  - `classificationConfidence`
  - `treatmentModality`
  - `delivery`
  - `ctGovUrl`
  - `sites`
  - `availableInstitutions`
  - `siteCount`
  - `inclusionCriteria`
  - `exclusionCriteria`
  - `primaryOutcomes`
  - `secondaryOutcomes`
  - `studyFirstPosted`
  - `lastUpdatePosted`
  - `lastSyncAt`
  - `pipelineVersion`
  - `sourceRun`

## 4. Why There Is An Importer

The site does not read the raw pipeline CSV or XLSX exports.

Instead, the website imports the website-shaped JSON through:

- `scripts/import_website_catalog.php`

This importer does several important things:

- validates the incoming catalog
- normalizes trial shape through the PHP backend
- preserves/updates catalog metadata
- writes the live `data/trials.json`
- removes forum threads/replies tied to trials no longer present
- writes an audit log entry
- restores the old catalog if the import fails mid-process

This means the importer is not just a file copy. It is the safety boundary between pipeline output and production website data.

## 6. Namecheap cPanel Side

The website is deployed to Namecheap shared hosting under the site document root, usually:

```text
/home/<cpanel_user>/public_html
```

The implementation assumes:

- PHP is available on the server
- `pdo_sqlite` is enabled
- `sqlite3` is enabled
- SSH is enabled in cPanel
- the PHP CLI is available for running the importer
- the hosting account can reach `clinicaltrials.gov` over the network

During implementation, those conditions were verified on the active Namecheap account.

### What Must Exist On The Server

The server needs the website code, not the full local development environment.

At minimum, the deployed site should include:

- `api/`
- `assets/`
- `config/`
- `css/`
- `data/`
- `js/`
- `scripts/`
- the `*-php.html` pages
- `.htaccess` if used by the site

The server does not need to run the Python pipeline itself during the weekly sync. GitHub does that.

### SSH Notes For Namecheap

Namecheap shared hosting SSH uses port `21098`.

The workflow explicitly configures:

- `HostKeyAlgorithms +ssh-rsa`
- `PubkeyAcceptedAlgorithms +ssh-rsa`

because Namecheap shared hosting still requires compatibility with older RSA SSH behavior on some clients/runners.

If someone removes that from the workflow, SSH may start failing again with handshake/host-key issues.

## 7. GitHub Actions Side

The weekly sync is defined in:

- `.github/workflows/weekly-sync.yml`

It currently:

1. checks out the repo
2. installs Python 3.11
3. installs pipeline dependencies
4. runs the pipeline with:
   - `--skip-excel`
   - `--skip-updates`
   - `--website-catalog-out build/website_trials.json`
5. uploads the catalog as a workflow artifact
6. configures SSH for Namecheap
7. tests SSH connectivity
8. uploads the generated catalog JSON via `scp`
9. runs the PHP importer remotely over SSH

### Schedule

The workflow currently runs:

- manually via `workflow_dispatch`
- weekly via cron at `17 9 * * 1`

That is Monday at `09:17 UTC`.

### GitHub Secrets

The workflow expects these repository secrets:

- `NAMECHEAP_HOST`
- `NAMECHEAP_PORT`
- `NAMECHEAP_USER`
- `REMOTE_APP_ROOT`
- `NAMECHEAP_SSH_KEY`
- `NAMECHEAP_SSH_KEY_B64`

`NAMECHEAP_SSH_KEY_B64` is preferred when valid.

If `NAMECHEAP_SSH_KEY_B64` is invalid, the workflow falls back to `NAMECHEAP_SSH_KEY`.

This fallback exists because multiline SSH secrets are easy to damage during copy/paste. Base64 is safer, but mistakes still happen.

## 8. Preferred SSH Secret Handling

The safest way to store the deploy key is:

```bash
gh secret set NAMECHEAP_SSH_KEY_B64 --body "$(openssl base64 -A -in ~/.ssh/<deploy_key>)"
```

This avoids manual clipboard corruption.

The workflow strips whitespace from `NAMECHEAP_SSH_KEY_B64` before decoding and validates the resulting key with:

```bash
ssh-keygen -y -f ~/.ssh/id_rsa
```

That validation was added because malformed GitHub secrets produced low-level `libcrypto` SSH errors that were otherwise hard to debug.

## 9. One-Time Deployment Flow

The weekly workflow only syncs the catalog.

It does not deploy the site code itself.

That means there is a one-time manual deployment step whenever:

- the site is first set up on Namecheap
- the PHP/JS/HTML code changes
- importer/backend/frontend changes need to reach production

Recommended deploy pattern:

- use `rsync` or `scp` over SSH from a trusted local machine
- deploy the application code to the document root
- do not overwrite live runtime data unintentionally

Be careful with:

- `data/trials.json`
- `data/physician_forum.sqlite`
- backup files in `data/`

Those should not be casually overwritten during routine code deploys.

## 10. Weekly Sync Flow

The current production sync path is:

1. GitHub Actions checks out this repository.
2. GitHub installs Python dependencies from `knowledgecenter/production_ready_pipeline/requirements.txt`.
3. GitHub runs the pipeline and creates `build/website_trials.json`.
4. GitHub uploads the catalog artifact for debugging/audit.
5. GitHub SSHes into Namecheap.
6. GitHub copies `build/website_trials.json` to the server home directory.
7. GitHub runs:

```bash
php '<REMOTE_APP_ROOT>/scripts/import_website_catalog.php' ~/website_trials.json
```

8. The importer validates and writes the live website catalog.
9. The temporary uploaded JSON file is deleted.

The importer is what makes the new data live.

## 11. Local Development And Testing

### Run The Website Locally

Use the PHP server, not a static file server:

```bash
php -S 127.0.0.1:8000
```

Then open:

- `http://127.0.0.1:8000/index-php.html`
- `http://127.0.0.1:8000/api/trials.php`

### Run The Pipeline Locally

Example smoke run:

```bash
python3 knowledgecenter/production_ready_pipeline/gu_pipeline.py --max-conditions 2 --page-size 20
```

Example website-only export:

```bash
python3 knowledgecenter/production_ready_pipeline/website_export.py --max-conditions 2 --out /tmp/website_trials.json
```

### Run Local End-To-End Sync

If the machine has both Python and PHP available, use:

```bash
scripts/sync_pipeline_to_website.sh
```

That script:

1. exports a website catalog to `/tmp/...json`
2. runs the PHP importer locally

This is mainly useful for local testing or non-Namecheap environments.

## 13. Known Design Decisions

### The website still uses `data/trials.json`

The project did not migrate the trial catalog into SQLite.

That was intentional. The existing site already used JSON for the catalog, so the lowest-risk integration path was:

- keep website catalog storage in JSON
- enrich the JSON schema
- let SQLite continue handling auth/community/forum

### GitHub runs the pipeline, not Namecheap

This was also intentional.

It avoids:

- Python version problems on shared hosting
- pipeline dependency management on Namecheap
- long-running jobs on a shared host

### The exporter sanitizes invalid emails

This was added after the first production-like workflow run failed on:

- `clinicaltrials.@hoag.org`

The Python exporter now removes invalid site/contact emails before the JSON reaches the PHP importer.

This matters because the PHP side correctly rejects invalid email values.

## 14. Troubleshooting

### GitHub workflow fails with SSH `exit code 255`

Look at the specific step:

- `Configure SSH for Namecheap`
- `Test SSH connectivity`
- `Upload catalog to Namecheap`
- `Import catalog on server`

Common causes:

- bad SSH secret formatting
- missing SSH key authorization in cPanel
- broken Namecheap RSA compatibility in SSH config
- wrong `REMOTE_APP_ROOT`

### SSH error mentions `libcrypto`

That usually means the private key secret is malformed.

Check the GitHub secret, especially `NAMECHEAP_SSH_KEY_B64`.

### SSH works locally but not in GitHub

Usually one of:

- the GitHub secret contains the wrong key
- the base64 secret was copied incorrectly
- the public key was not authorized in cPanel

### Import fails with validation errors

The importer validates every row. If one row is bad, the import fails.

Typical causes:

- invalid email format
- invalid date format
- invalid cancer type normalization

Check the workflow log for:

```text
Catalog import failed validation:
- Row X: ...
```

Then inspect the exported JSON artifact from the same workflow run.

### The site does not reflect the latest sync

Check:

1. did the workflow finish successfully
2. did the importer run successfully
3. does `api/trials.php` show updated `metadata.lastSyncAt`
4. was the site code actually deployed to the same `REMOTE_APP_ROOT`

## 15. If You Need To Change The Schema

Any schema change must be thought of as a cross-cutting change across:

1. pipeline exporter
2. PHP normalization/validation
3. API response contract
4. frontend rendering

The relevant files are:

- `knowledgecenter/production_ready_pipeline/website_export.py`
- `api/bootstrap.php`
- `api/trials.php`
- `js/utils.js`
- `js/trial-manager-php.js`
- `js/main-php.js`
- `trial-detail-php.html`

Do not change only one of those layers and assume the system will stay consistent.

## 16. Handoff Checklist

If a new engineer is taking over, they should verify these first:

1. GitHub Actions workflow exists and is green.
2. GitHub repository secrets are present.
3. Namecheap cPanel SSH is enabled.
4. The deploy key is still authorized in cPanel.
5. The site code exists at the configured `REMOTE_APP_ROOT`.
6. `api/trials.php` returns `metadata`, `trials`, and `count`.
7. The latest workflow artifact contains a valid `website_trials.json`.
8. The weekly cron schedule is still the desired one.

## 17. Recommended Future Improvements

- Add a dedicated PHP lint/test step in CI for environments where PHP is available.
- Add a pre-import validation summary step to the GitHub workflow.
- Add a post-import smoke check against `api/trials.php`.
- Add notification on workflow failure.
- Add a visible "last sync" UI element that reads from catalog metadata.
- Consider pinning current GitHub action SHAs if stricter supply-chain controls are needed.

## 18. Short Version

If you only remember five things, remember these:

1. The website reads `data/trials.json`, not raw pipeline exports.
2. GitHub Actions runs the pipeline because Namecheap shared hosting is not the right place for it.
3. Namecheap only imports the generated catalog through `scripts/import_website_catalog.php`.
4. `.github/workflows/weekly-sync.yml` is the operational sync path.
5. Schema changes must be updated across exporter, PHP backend, API, and frontend together.
